# Changelog

## 0.10.6

This release ships work that merged to `main` after the `0.10.5` tag was cut: the `protect --grant` AAP gate (#179) and the NanoMind classifier router gate (#180), plus the #185 security dep bump. `0.10.5` published the bundled-hackmyagent `0.23.6` bump and the `scan-soul --explain` passthrough; the NanoMind classifier *adapter* landed in `0.10.5` but was not wired into the router (it shipped dormant). `0.10.6` is the release that actually delivers the natural-language trust-boundary gate to installed CLIs.

### Features
- **`opena2a protect --grant <grant-ref> --atx <path>`: opt-in Agent Authorization Protocol gate** ([#179](https://github.com/opena2a-org/opena2a/pull/179)). Before any scan, `protect` presents an ATX to the local Secretless broker over its Unix socket and proceeds only if the broker authorizes the grant. New `packages/cli/src/aap/` module is the first TypeScript AAP consumer (shape mirrors agent-identity-management's Python `grant_client.py`); behavior is unchanged when `--grant` is not set. Exit codes: 0 (authorized), 2 (`--grant` without `--atx` or invalid ATX), 3 (403 opaque denial, AAP §6.6), 4 (broker socket unreachable), 5 (unexpected error), 6 (broker returned a non-403 non-200 status -- body never echoed to the user). Hardening: default-socket connections require socket-uid == process-uid (same-box impostor-broker defense); ATX file capped at 256 KiB; response body capped at 1 MiB; ANSI/C0 control chars in the user-supplied grant reference are stripped before any stderr output. Defends T-3002 / T-3003 / T-3006 / T-8002 at the CLI surface.
- **NanoMind classifier gate at the natural-language trust boundary** ([#180](https://github.com/opena2a-org/opena2a/pull/180)). `handleNaturalLanguage` (`packages/cli/src/natural/llm-fallback.ts`) now runs the NanoMind classifier FIRST, before any static intent matching or LLM fallback. Adversarial free-form input classified as `prompt_injection`, `tool_misuse`, `exfiltration_pattern`, or `data_extraction` at confidence > 0.8 is refused before it can be translated into a command; the function returns a structured `NaturalLanguageBlock` and the CLI renders a CISO-readable block message (observation + verdict + remediation) through `@opena2a/cli-ui` (`verdictColor`, `divider`). The classifier is defense-in-depth: a `null` result (daemon unreachable, timeout, schema violation, SSRF rejection) means "not blocked" and the existing intent flow runs unchanged -- a missing daemon never errors or logs (silent-fallback contract). The block message renders only the canonical `attackClass` enum + confidence + sanitized model version; the user's adversarial input and the daemon's `evidence`/`remediation` are never echoed back. Remediation points to the always-available escape hatch -- running the intended command explicitly -- rather than a `--no-classify` flag (deferred to a later release, so the message cites no unregistered flag). New tests: `__tests__/natural/router-guard.test.ts` (8 cases: blocked-short-circuits-before-`matchIntent`, benign / below-threshold / daemon-unreachable proceed paths, evidence/remediation never leak with real ESC/BEL bytes, and the pure block-message builder).

## Earlier work shipping in 0.10.5

### Features
- **NanoMind classifier adapter for the natural-language layer.** New module `packages/cli/src/natural/nanomind-classifier.ts` plus types at `nanomind-types.ts` ports the aicomply 2.0 reference adapter to opena2a-cli. Speaks the frozen `@nanomind/daemon` v0.3.0 wire contract: HTTP POST to `127.0.0.1:47200/v1/infer`, strict schema validation, block rule `attackClass !== '' && confidence > 0.8` per AIM FGA Step 5. Returns a simple `{ blocked, attackClass, confidence, modelVersion, latencyMs } | null` shape; null on any failure (network, non-2xx, malformed, timeout) so callers silently fall back. Daemon `evidence` and `remediation` are deliberately dropped in the mapper because they can carry attacker-influenced bytes. Liveness probe `isNanoMindDaemonAvailable()` against `/health`. No router wiring in this drop; that lands in a follow-up. `MOCK_NANOMIND_URL` env honored for tests.

### Security
- **Daemon communication restricted to loopback only.** Both `classifyWithNanoMindDaemon` and `isNanoMindDaemonAvailable` reject any `baseUrl` that is not `http(s)://(127.0.0.1|localhost|::1|[::1]):port` BEFORE issuing the request, so a misconfigured `MOCK_NANOMIND_URL` (or a future caller passing a malicious `baseUrl`) cannot exfiltrate user input to an external endpoint. Rejection returns `null` rather than throwing, preserving the silent-fallback contract.
- **Response-body size cap (1 MiB) on `/v1/infer`.** A compromised or buggy daemon could return a multi-gigabyte body and OOM the caller; the adapter fast-rejects on `Content-Length > MAX_NANOMIND_RESPONSE_BYTES` and re-checks the actual body length post-read. Defense-in-depth against a hostile process bound to the loopback port.
- 32 vitest tests pin the full contract: empty / whitespace short-circuit, blocked / not-blocked at the strict-`>`-0.8 threshold, schema rejection on missing fields / unknown enum / out-of-range confidence / negative latencyMs / malformed JSON / non-2xx, timeout abort with lower-bound assertion, trust-boundary (ANSI + shell-payload bytes in `evidence` / `remediation` MUST NOT leak into the returned shape), SSRF refusal on non-loopback URL / non-http scheme / malformed URL, response-size cap via declared `Content-Length` AND actual body length, `mapInferResponseToClassification` pure-mapper behavior, `isLocalhostHttpUrl` allow / reject coverage, and the `MOCK_NANOMIND_URL` env override.

## 0.10.2

### Security
- **OAuth tokens now stored in the OS keychain instead of plaintext on disk.** `opena2a login` previously wrote `accessToken` and `refreshToken` to `~/.opena2a/auth.json` with mode `0600`. Filesystem permissions kept the tokens away from other users on the box, but any process running as the user — Claude Code, an IDE plugin, an MCP server, an npm postinstall, a stolen laptop with FileVault unlocked — could read the tokens for the entire refresh-token lifetime. Tokens now go to macOS Keychain (via `security`) on Darwin and to the Linux Secret Service (via `secret-tool` / libsecret) on Linux. Service name `opena2a-cli`, account = `${serverUrl}:access` / `${serverUrl}:refresh`, label `opena2a-cli: <serverUrl> (access|refresh)` — visible as a discrete entry in macOS Passwords.app. The metadata file at `~/.opena2a/auth.json` (mode `0600`) is retained and now carries `serverUrl`, `expiresAt`, `tokenType`, `authenticatedAt`, and a new `tokenStorage: 'keychain' | 'file'` discriminator. When `tokenStorage === 'keychain'`, the file does NOT contain `accessToken` or `refreshToken` — they live in the keychain only. New helper `packages/cli/src/util/keychain.ts` (`MacOSKeychain`, `LinuxKeychain`, `NullKeychain` with platform dispatch via `getKeychain()`). 16 new unit tests in `__tests__/util/keychain.test.ts` and `__tests__/util/auth.test.ts` (round-trip / mode-0600 / removeAuth / migration / malformed-file / missing-token / keychain-empty / mock-backend no-token-leak / orphan-clearing logout / `validateServerUrl` paths). Test override: set `OPENA2A_AUTH_FORCE_FILE=1` to force the file fallback.
- **Transparent migration for upgraders.** Pre-keychain `~/.opena2a/auth.json` files (tokens inline, no `tokenStorage` field) are detected and rewritten on the first command after upgrade. The CLI prints `Note: tokens migrated to OS keychain. ~/.opena2a/auth.json now holds metadata only.` once. Idempotent; no user action required.
- **Headless / Docker / Windows fallback.** When the OS keychain is unavailable (Windows today, headless Linux without DBus, locked keychain prompts), tokens fall back to the `0600` file with a one-line stderr warning at login. Repeat saves on a known file-fallback host stay silent (token-refresh inside `tryRefreshAuth` no longer spams stderr). Windows native `cmdkey` integration is tracked separately in `todo/2026-04-30-cli-auth-keychain-storage-P1.md`.
- **`opena2a logout` now enumerates and clears every `opena2a-cli` keychain entry, not just the one matching the current `serverUrl`.** Catches server-switch orphans (user logged into cloud, then into self-hosted without logging out — the cloud refresh token would otherwise persist in the keychain indefinitely) AND clears the keychain even when the metadata file is malformed. Also deletes the metadata file. New `KeychainBackend.listAccounts()` method (macOS via `security dump-keychain`, Linux via `secret-tool search --all`).
- **`validateServerUrl()` defense layer in `keychain.ts`.** Rejects empty strings, leading hyphens, ASCII control characters (NUL / LF / CR), and overlength inputs before they reach `security` / `secret-tool` argv. `execFileSync` already prevents shell-level injection, but a poisoned `serverUrl` containing NUL would truncate on libsecret, and embedded LF would split account strings on enumeration — defense in depth.

Reduces the click-distance from "agent registered" to "I can see my agent" from N+1 clicks to zero.

- `main()` wraps `program.parseAsync` in a try/catch/finally so `tele.error()` fires on subcommand throws and `tele.flush()` always runs before exit. Subcommand tracking happens via Commander `preAction`/`postAction` hooks.
- CommonJS / ESM bridge: the runtime imports of `@opena2a/telemetry` and `@opena2a/cli-ui` use dynamic `import()` inside the async `main()`, with `import type ... with { 'resolution-mode': 'import' }` for the TS type-only reference. Same pattern shipped in DVAA 0.8.2 and secretless-ai 0.16.0.

## 0.8.25

### Changed
- **Trust queries route through `@opena2a/registry-client@0.1.0` (exact pin).** Four call sites migrated via dynamic `await import(...)` for CJS compatibility: `util/registry-enrichment.ts` (batch), `util/report-submission.ts` (publish), `commands/verify.ts` (profile), `commands/mcp-audit.ts` (score). All three fleet CLIs now share one HTTP client implementation (published with SLSA v1 provenance). Per CA-034 M1. No user-visible output change; registry returns the same canonical trust data either way.
- `commands/verify.ts` tamper-detection fetch stays as a direct call: `/api/v1/trust/query?name=...&hash=...` uses a `hash` parameter not exposed by `@opena2a/registry-client@0.1.0`. Documented inline; revisit when the client adds a hash option.

## 0.8.24

### Bug Fixes
- Scoring: removed redundant 8-point deduction for missing `.gitignore` in the configuration category. HackMyAgent already covers this at LOW severity; the overlay was double-counting and caused `opena2a scan` to disagree with `hackmyagent secure` on the same target (95 MEDIUM vs 98 LOW). Users will see the same score climb from 95 to 98 on the reference fixture. The `.gitignore` hygiene check itself still reports `warn` when missing, so the item remains visible in the checks list.

## 0.6.3

### Breaking Changes
- `scanIdentity()` now checks for `.opena2a/aim/identity.json` instead of just the `.opena2a/` directory existing. Projects with a bare `.opena2a/` directory (no identity file inside) will now correctly report "no identity" instead of "identity initialized."

### New Features
- Shadow AI detection (`opena2a detect`) with governance scoring
- HTML executive report (`--report`)
- CSV asset inventory export (`--export-csv`)
- Shadow AI phase added to `opena2a review`
- Community contribution flow for detect scans and trust scores

### Bug Fixes
- `identity create` no longer crashes with "getOrCreateIdentity is not a function"
- `identity list` no longer auto-creates an identity when none exists
- `identity policy load` correctly parses YAML with `effect` field
- `identity check` correctly evaluates policy rules
- SOUL.md no longer flagged as "grants broad permissions" in AI config scan
