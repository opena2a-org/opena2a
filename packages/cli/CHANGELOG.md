# Changelog

## Unreleased

### Fixes
- **`review` no longer penalizes a clean project for not adopting opt-in OpenA2A tooling.** A clean repo scored ~76 ("improving") because three composite dimensions docked it for *un-adopted* features rather than real risk: Config Integrity gave `50` for unsigned configs, Shield posture sat at its baseline (~40–55) when `shield init` hadn't been run, and Shadow-AI governance dropped for "no registered identity / no SOUL.md" and for ambient AI tools detected on the developer's machine (`ps aux`). None of these mean the scanned **target** is dangerous. Per [CHIEF-CDS], adoption is now a **recovery opportunity, not a composite penalty**: unsigned configs score neutral-high (signing remains a recovery nudge; only *tampering* is penalized), and Shield / governance contribute a penalty to the composite only when a genuine target-risk signal fired (a classified critical/high Shield runtime finding; a target-local critical MCP/AI-config — the same signal the dominant-analyzer floor uses). A clean project now scores in the "good" band (≥ 80). To keep this from diluting a real exposure, a confirmed **critical credential** drives the dominant-analyzer floor (a committed production key pulls the verdict to "needs attention", not a "good 86"). New `shieldCompositeScore` / `governanceCompositeScore` unit tests and a ≥80 clean-project regression guard. (Note: the `repo/buggy/leaky-env-example` corpus fixture is a deliberately-subtle misconfig — `FAKE`-marked placeholder values + standard "rename `.env.example` to `.env`" onboarding — that is indistinguishable from good practice at the repo-composite level; separating it from benign needs a corpus-side fixture change, tracked separately, not a scanner heuristic that would false-positive on normal `.env.example` usage.)
- **`review` composite verdict now agrees in direction with its harshest analyzer** ([#175](https://github.com/opena2a-org/opena2a/issues/175)). On a kitchen-sink fixture, `opena2a review` reported `67/100 "improving"` (recoverable) while `opena2a check` reported `0/100` with 34 critical findings — a direction disagreement that could lead a fresh user to conclude a fully-malicious project was salvageable. The composite is a weighted average, so HMA `secure` (0/100) was out-voted by the clean Project Scan / Credentials / Config Integrity dimensions. A **dominant-analyzer floor** ([CHIEF-CDS]) now clamps the composite down to the lowest analyzer score whenever any *target-malice* analyzer lands in the critical band (`< 30`). Participants are scoped to analyzers whose low score unambiguously means the **target** is dangerous: Project Scan, Credentials, Config Integrity, HMA Scan (only when it ran), and Project Governance. The last is the **target-local** slice of governance only — in-repo critical MCP-server declarations and critical AI configs — *not* the raw `governanceScore`, which is partly derived from a host process scan (`ps aux`) and so reflects AI tools running on the developer's machine, not the scanned project. **Shield Analysis and the raw Shadow-AI governance score are deliberately excluded** from the floor: both can fall into the critical band from an unconfigured developer environment (Shield's baseline-25 posture; host agents in the process scan) rather than target malice, which would wrongly clamp a clean repo. Non-finite analyzer scores are filtered so a malformed HMA payload cannot silently disable the floor. Kitchen-sink now correctly reports `0/100 needs-attention`; benign and buggy-recoverable fixtures are unaffected, and a clean repo on a machine running ungoverned AI tools is no longer downgraded. Because the floor's deepest signal is HMA, when HMA did not run (`--skip-hma` or not installed) `review` emits a **provisional-verdict notice** to stderr and sets `provisional: true` in the `--json` report — so neither a human nor a CI consumer mistakes a non-floored score for a clean one. New `buildFloorParticipants` / `targetGovernanceFloorScore` / `applyDominantAnalyzerFloor` regression tests, plus a `review()` integration test for the provisional path, in `__tests__/commands/review.test.ts`.

## 0.10.10

### Added
- **`opena2a comply` — inline compliance classifier.** A new top-level command that classifies content for PII, credentials, and regulated data before an agent forwards it to a hosted LLM. Detection and verdict are delegated to the `@opena2a/aicomply` engine (the shared source of truth); the command collects sources (files or stdin) and renders the verdict in opena2a house-style. Output is always masked — the raw detected value never reaches stdout or the JSON output, even though the command's whole purpose is to keep sensitive data out of an LLM. The mask is fail-safe for short values: anything 8 chars or shorter is masked entirely (a fixed-length head would otherwise reveal most of a short password / token), and longer values reveal at most a 3-char head + 2-char tail. Untrusted strings (file labels, finding types) are passed through the cli-ui terminal sanitizer so a crafted value cannot inject control bytes. Inputs are bounded at 5 MiB and over-limit / unreadable / IO-error / internal-failure paths all fail **closed** (exit 2 — never a misleading CLEAN or an OOM crash). Usable as a CI gate: exit `0` CLEAN, `1` VIOLATION/DENY, `2` usage error. Supports `--json`, `--quiet`, and stdin (`echo "…" | opena2a comply`). On a CLEAN verdict the command still states which classification layers ran and how to enable the semantic Guard layer (CISO Rule 11 — no dead ends). Python users can reach the same engine with `pip install aicomply`. New `__tests__/commands/comply.test.ts` (9 tests, including mask-integrity and oversize-input assertions) and a built-binary dispatch smoke (`npm run release-smoke:comply`).

### Fixes
- **`protect` / `review` / `init` detect a short `sk-ant-api03-` Anthropic key** ([#184](https://github.com/opena2a-org/opena2a/issues/184)). The `CRED-001` pattern required an 80+ char body, so a realistic short/fake key (`sk-ant-api03-FAKE_…`, 28 chars) was silently skipped while the OpenAI key on the very next line was flagged — an asymmetric coverage gap exactly where users expect the strongest coverage. The `sk-ant-api\d{2}-` prefix is highly specific to Anthropic, so the body floor is lowered to 20 chars (matching the OpenAI `CRED-002` floor). A bare prefix with no body is still not flagged (no false positives), and full-length production keys are unaffected. Regression tests in `__tests__/util/credential-label-classification.test.ts`.
- **The contribute prompt no longer corrupts machine-readable output.** `check`/`review`/`detect`/`identity` wrote an interactive "Your scans help the community…" prompt to stderr after enough scans, so `opena2a check --json 2>&1 | jq` merged the banner into the parsed stream and broke it. `recordScanAndMaybePrompt` now suppresses the prompt in machine-readable mode (`--json` / `--format json|sarif` / `--ci`) and whenever stdout is not a TTY; the scan count is still recorded so the prompt fires on a later interactive run. Surfaced by the 0.10.8 fresh-user release test.
- **`init` empty-project Verify command no longer errors.** The `ENV-DOTENV` finding (".env not in .gitignore") suggested `cat .gitignore | grep -c '.env'`, which fails with "No such file or directory" on a project that has no `.gitignore`. The Verify command is now `grep -c '\.env' .gitignore 2>/dev/null || echo 0`, which prints `0` (correctly reading as ".env not ignored") and exits cleanly when `.gitignore` is absent. New `__tests__/commands/init-verify-command.test.ts`. Surfaced by the 0.10.8 fresh-user release test.

### Fixes (first shipped here — were merged after the stale 0.10.8 publish)
- **`protect` / `review` / `init` no longer mislabel every `sk-` credential as "OpenAI API Key".** The local catch-all pattern (`CRED-002`) matched any non-Anthropic `sk-` token and surfaced it as `OpenAI API Key` with an `OPENAI_API_KEY` env var — wrong for OpenRouter (`sk-or-v1-…`) and any future `sk-`-prefixed provider, and the generic-assignment bucket (`CRED-004`) labelled Stripe keys (`sk_live_…` / `sk_test_…`) as a generic "API Key". The displayed label and suggested env var for those two generic buckets are now routed through the canonical `@opena2a/credential-patterns` catalog, which orders specific prefixes before catch-alls, so the precise provider is recovered (Anthropic / OpenAI Project / OpenAI Legacy / OpenRouter / Stripe …). Detection is unchanged — only the label of values the scanner already flagged is refined; the deliberate "(Gemini drift risk)" / "(Bedrock drift risk)" framing on the Google/AWS patterns is preserved. The ESM-only catalog is loaded once per scan via a dynamic `import()` (this CLI is CommonJS), with a graceful fallback to the local label if it cannot be loaded. New regression test `__tests__/util/credential-label-classification.test.ts` covers `sk-ant-…`, OpenRouter, OpenAI, and Stripe `sk_live_…` / `sk_test_…` fixtures.
- **The `Why:` / `Impact:` rationale no longer contradicts a refined credential label.** When the refined `Type` named a different provider than the local pattern (e.g. `OpenRouter API Key`), the explanation still read "OpenAI API key … grants full OpenAI API access", and Stripe keys kept the generic "Generic API key found in a variable assignment" copy. `refineCredentialLabel` now also swaps to provider-neutral prose keyed off the refined title whenever the title changes, so the displayed `Why:`/`Impact:` always agree with the label (the catalog carries no prose, so neutral-but-correct is preferred over inventing provider-specific copy). Non-refined findings (Anthropic, GitHub, AWS/Google drift) keep their richer local copy verbatim. Caught by the 0.10.8 fresh-user release test.

## 0.10.7

Resolves three follow-ups surfaced by the `0.10.6` release test (#188, #189, #190). No bundled-tool version change; the `release-smoke:corpus` goldens are unchanged.

### Fixes
- **`opena2a login --ci` fails fast instead of hanging** ([#189](https://github.com/opena2a-org/opena2a/issues/189)). The OAuth 2.0 device-code flow needs a browser to approve the user code, which does not exist in `--ci` / non-interactive environments, so `login --ci` previously blocked on `Waiting for authentication...` until the device code expired (or the CI runner timed out). In `--ci` mode `login` now exits non-zero immediately, after the already-authenticated short-circuit (a CI job with cached valid credentials still returns 0) and before any device-code request reaches the network. The message points at the non-interactive path: pass `--api-key` to server commands, or run `opena2a login` once on a workstation to cache credentials. JSON mode emits `{"error":"interactive_required",...}`. New regression test `__tests__/commands/login-ci.test.ts`.
- **Next-step suggestions use the `opena2a`-prefixed form, not bundled tool names** ([#190](https://github.com/opena2a-org/opena2a/issues/190)). A user who installed `opena2a` was sometimes told to run a bundled backend binary they may not know exists (`hackmyagent secure`, `ai-trust check`, `npx ai-trust --help`). The fix has two parts:
  - opena2a-cli's own emitted strings now cite the `opena2a` command (`opena2a secure`, `opena2a scan`, `opena2a registry`): `trust` Action line, `claim` next step, `init` / `identity` / `shield status` setup hints, `review` action column, and the adapter `--help` header (which no longer prints `Run \`npx <tool> --help\``).
  - Delegated stdout from spawn adapters (the `registry` / `publish` ai-trust path) is now rebranded line-buffered as it streams, so `Usage: ai-trust check ...` and next-steps like `ai-trust check <pkg>` render as `opena2a registry ...`. Only commands with a faithful `opena2a` equivalent are rewritten (a bundled subcommand opena2a-cli does not expose 1:1, e.g. `ai-trust audit`, is left unchanged rather than rewritten to a command that would not resolve). Rebranding never runs in `--json` / `--format json|sarif` mode, and the captured stdout returned to callers is unmodified. Registry data (the ATP `actionRequired` field) is rebranded at render time. New `src/util/rebrand.ts` with `__tests__/util/rebrand.test.ts`. `AI_TRUST_CLI_PREFIX` / `CRYPTOSERVE_CLI_PREFIX` are also exported to spawned tools for forward-compat (mirrors `HMA_CLI_PREFIX`), so the rebrand becomes a no-op once those tools honor the prefix natively.

### Internal
- **HackMyAgent self-scan of `packages/cli` is clean via a documented `.hmaignore`** ([#188](https://github.com/opena2a-org/opena2a/issues/188)). All ten findings on opena2a-cli's own source (NEMO-007 x5, NEMO-008 x2, NEMO-009 critical, MEM-006 x2) were triaged as false positives on this tool's own subprocess / test / report-array code and suppressed with per-entry rationale; no detection was weakened and no source was contorted to dodge a heuristic. Test fixtures and dev scripts are suppressed by path (`__tests__`, `scripts`); the subprocess launchers by path (`src/adapters`, `src/shield/llm-backend.ts`); and the heuristic-misfire files by exact path (`src/commands/shield.ts`, `src/commands/guard.ts`, `src/commands/guard-snapshots.ts`) rather than a project-wide check-ID, so a genuine future NEMO-008/MEM-006 elsewhere still surfaces. `.hmaignore` is not in the npm `files` array, so it does not ship. (Deferred follow-ups: cryptoserve non-TTY banner suppression, which needs cryptoserve installed to verify; and a pre-existing `opena2a registry <pkg> --json` injecting `--format json`, which ai-trust rejects.)

## 0.10.6

This release ships work that merged to `main` after the `0.10.5` tag was cut: the `protect --grant` AAP gate (#179) and the NanoMind classifier router gate (#180), plus the #185 security dep bump. `0.10.5` published the bundled-hackmyagent `0.23.6` bump and the `scan-soul --explain` passthrough; the NanoMind classifier *adapter* landed in `0.10.5` but was not wired into the router (it shipped dormant). `0.10.6` is the release that actually delivers the natural-language trust-boundary gate to installed CLIs.

### Features
- **`opena2a protect --grant <grant-ref> --atx <path>`: opt-in Agent Authorization Protocol gate** ([#179](https://github.com/opena2a-org/opena2a/pull/179)). Before any scan, `protect` presents an ATX to the local Secretless broker over its Unix socket and proceeds only if the broker authorizes the grant. New `packages/cli/src/aap/` module is the first TypeScript AAP consumer (shape mirrors agent-identity-management's Python `grant_client.py`); behavior is unchanged when `--grant` is not set. Exit codes: 0 (authorized), 2 (`--grant` without `--atx` or invalid ATX), 3 (403 opaque denial, AAP §6.6), 4 (broker socket unreachable), 5 (unexpected error), 6 (broker returned a non-403 non-200 status -- body never echoed to the user). Hardening: default-socket connections require socket-uid == process-uid (same-box impostor-broker defense); ATX file capped at 256 KiB; response body capped at 1 MiB; ANSI/C0 control chars in the user-supplied grant reference are stripped before any stderr output. Defends T-3002 / T-3003 / T-3006 / T-8002 at the CLI surface.
- **NanoMind classifier gate at the natural-language trust boundary** ([#180](https://github.com/opena2a-org/opena2a/pull/180)). `handleNaturalLanguage` (`packages/cli/src/natural/llm-fallback.ts`) now runs the NanoMind classifier FIRST, before any static intent matching or LLM fallback. Adversarial free-form input classified as `prompt_injection`, `tool_misuse`, `exfiltration_pattern`, or `data_extraction` at confidence > 0.8 is refused before it can be translated into a command; the function returns a structured `NaturalLanguageBlock` and the CLI renders a CISO-readable block message (observation + verdict + remediation) through `@opena2a/cli-ui` (`verdictColor`, `divider`). The classifier is defense-in-depth: a `null` result (daemon unreachable, timeout, schema violation, SSRF rejection) means "not blocked" and the existing intent flow runs unchanged -- a missing daemon never errors or logs (silent-fallback contract). The block message renders only the canonical `attackClass` enum + confidence + sanitized model version; the user's adversarial input and the daemon's `evidence`/`remediation` are never echoed back. Remediation points to the always-available escape hatch -- running the intended command explicitly -- rather than a `--no-classify` flag (deferred to a later release, so the message cites no unregistered flag). New tests: `__tests__/natural/router-guard.test.ts` (8 cases: blocked-short-circuits-before-`matchIntent`, benign / below-threshold / daemon-unreachable proceed paths, evidence/remediation never leak with real ESC/BEL bytes, and the pure block-message builder).

### Fixes
- **`--help` no longer advertises a natural-language example that collides with a real command.** The "Smart Features" block listed `opena2a detect credentials` as "Natural language command matching", but `detect` is a registered command taking `[directory]`, so the phrase scanned a `credentials/` directory (`Cannot access directory: .../credentials`) instead of matching an intent -- and shell quoting could not rescue it, since the example's first token is what decides command-vs-NL routing. Replaced with `opena2a audit my project` (first token `audit` is not a command; resolves via the NL matcher to `opena2a scan secure`). Extracted the core command-name list into `src/natural/known-commands.ts` as the single source of truth shared by the dispatch fallback and a new regression test (`__tests__/natural/help-nl-examples.test.ts`) that asserts every documented NL example routes through the matcher, not a command. Found by the 0.10.6 fresh-user release test.

## Earlier work shipping in 0.10.5

### Features
- **NanoMind classifier adapter for the natural-language layer.** New module `packages/cli/src/natural/nanomind-classifier.ts` plus types at `nanomind-types.ts` ports the aicomply 2.0 reference adapter to opena2a-cli. Speaks the frozen `@nanomind/daemon` v0.3.0 wire contract: HTTP POST to `127.0.0.1:47200/v1/infer`, strict schema validation, block rule `attackClass !== '' && confidence > 0.8` per AIM FGA Step 5. Returns a simple `{ blocked, attackClass, confidence, modelVersion, latencyMs } | null` shape; null on any failure (network, non-2xx, malformed, timeout) so callers silently fall back. Daemon `evidence` and `remediation` are deliberately dropped in the mapper because they can carry attacker-influenced bytes. Liveness probe `isNanoMindDaemonAvailable()` against `/health`. No router wiring in this drop; that lands in a follow-up. `MOCK_NANOMIND_URL` env honored for tests.

### Security
- **Daemon communication restricted to loopback only.** Both `classifyWithNanoMindDaemon` and `isNanoMindDaemonAvailable` reject any `baseUrl` that is not `http(s)://(127.0.0.1|localhost|::1|[::1]):port` BEFORE issuing the request, so a misconfigured `MOCK_NANOMIND_URL` (or a future caller passing a malicious `baseUrl`) cannot exfiltrate user input to an external endpoint. Rejection returns `null` rather than throwing, preserving the silent-fallback contract.
- **Response-body size cap (1 MiB) on `/v1/infer`.** A compromised or buggy daemon could return a multi-gigabyte body and OOM the caller; the adapter fast-rejects on `Content-Length > MAX_NANOMIND_RESPONSE_BYTES` and re-checks the actual body length post-read. Defense-in-depth against a hostile process bound to the loopback port.
- 32 vitest tests pin the full contract: empty / whitespace short-circuit, blocked / not-blocked at the strict-`>`-0.8 threshold, schema rejection on missing fields / unknown enum / out-of-range confidence / negative latencyMs / malformed JSON / non-2xx, timeout abort with lower-bound assertion, trust-boundary (ANSI + shell-payload bytes in `evidence` / `remediation` MUST NOT leak into the returned shape), SSRF refusal on non-loopback URL / non-http scheme / malformed URL, response-size cap via declared `Content-Length` AND actual body length, `mapInferResponseToClassification` pure-mapper behavior, `isLocalhostHttpUrl` allow / reject coverage, and the `MOCK_NANOMIND_URL` env override.

## 0.10.2

### Security
- **OAuth tokens now stored in the OS keychain instead of plaintext on disk.** `opena2a login` previously wrote `accessToken` and `refreshToken` to `~/.opena2a/auth.json` with mode `0600`. Filesystem permissions kept the tokens away from other users on the box, but any process running as the user — Claude Code, an IDE plugin, an MCP server, an npm postinstall, a stolen laptop with FileVault unlocked — could read the tokens for the entire refresh-token lifetime. Tokens now go to macOS Keychain (via `security`) on Darwin and to the Linux Secret Service (via `secret-tool` / libsecret) on Linux. Service name `opena2a-cli`, account = `${serverUrl}:access` / `${serverUrl}:refresh`, label `opena2a-cli: <serverUrl> (access|refresh)` — visible as a discrete entry in macOS Passwords.app. The metadata file at `~/.opena2a/auth.json` (mode `0600`) is retained and now carries `serverUrl`, `expiresAt`, `tokenType`, `authenticatedAt`, and a new `tokenStorage: 'keychain' | 'file'` discriminator. When `tokenStorage === 'keychain'`, the file does NOT contain `accessToken` or `refreshToken` — they live in the keychain only. New helper `packages/cli/src/util/keychain.ts` (`MacOSKeychain`, `LinuxKeychain`, `NullKeychain` with platform dispatch via `getKeychain()`). 16 new unit tests in `__tests__/util/keychain.test.ts` and `__tests__/util/auth.test.ts` (round-trip / mode-0600 / removeAuth / migration / malformed-file / missing-token / keychain-empty / mock-backend no-token-leak / orphan-clearing logout / `validateServerUrl` paths). Test override: set `OPENA2A_AUTH_FORCE_FILE=1` to force the file fallback.
- **Transparent migration for upgraders.** Pre-keychain `~/.opena2a/auth.json` files (tokens inline, no `tokenStorage` field) are detected and rewritten on the first command after upgrade. The CLI prints `Note: tokens migrated to OS keychain. ~/.opena2a/auth.json now holds metadata only.` once. Idempotent; no user action required.
- **Headless / Docker / Windows fallback.** When the OS keychain is unavailable (Windows today, headless Linux without DBus, locked keychain prompts), tokens fall back to the `0600` file with a one-line stderr warning at login. Repeat saves on a known file-fallback host stay silent (token-refresh inside `tryRefreshAuth` no longer spams stderr). Windows native `cmdkey` integration is tracked separately in `todo/2026-04-30-cli-auth-keychain-storage-P1.md`.
- **`opena2a logout` now enumerates and clears every `opena2a-cli` keychain entry, not just the one matching the current `serverUrl`.** Catches server-switch orphans (user logged into cloud, then into self-hosted without logging out — the cloud refresh token would otherwise persist in the keychain indefinitely) AND clears the keychain even when the metadata file is malformed. Also deletes the metadata file. New `KeychainBackend.listAccounts()` method (macOS via `security dump-keychain`, Linux via `secret-tool search --all`).
- **`validateServerUrl()` defense layer in `keychain.ts`.** Rejects empty strings, leading hyphens, ASCII control characters (NUL / LF / CR), and overlength inputs before they reach `security` / `secret-tool` argv. `execFileSync` already prevents shell-level injection, but a poisoned `serverUrl` containing NUL would truncate on libsecret, and embedded LF would split account strings on enumeration — defense in depth.

### Bug Fixes
- **`opena2a setup` printed the backend host as the dashboard URL.** The success block built `${auth.serverUrl}/dashboard` directly, which on AIM Cloud became `https://aim.oa2a.org/dashboard` — the API host, which serves a JSON health endpoint, not the UI. Backend = `oa2a.org`, frontend = `opena2a.org`; the two were never meant to be the same host. New helper `resolveDashboardUrl(serverUrl)` in `packages/cli/src/util/server-url.ts` maps API host → frontend host (`aim.oa2a.org` → `aim.opena2a.org`, `api.aim.opena2a.org` → `aim.opena2a.org`, self-hosted hostnames pass through unchanged). 6 new unit tests in `__tests__/util/server-url.test.ts` pin the four URL flavors plus a defensive non-URL case. Release-smoke section 4 (`docs/testing/release-smoke.md`) asserts the wire output every release — URL drift is invisible to scoring tests so it needs its own gate.
- **`opena2a setup` deep-links to the agent that was just created.** Printed link is now `https://aim.opena2a.org/dashboard/agents/<agentId>` (the resource the user just created), with a separate `MCP inventory:` line when MCP servers were attached. Reduces the click-distance from "agent registered" to "I can see my agent" from N+1 clicks to zero.
- **`opena2a setup` auto-launches login when not authenticated.** Previously emitted `Run: opena2a login` and exited code 1, which forced the user to cancel + retry the setup flow. Setup now calls into the existing `login()` flow inline, prints a one-line "Press Ctrl+C to cancel and use --server <url>" hint for users who actually want a self-hosted AIM, and continues into identity creation once auth lands. JSON callers (`--json`) keep the strict-mode behavior — they still error out with the machine-readable code.
- **`opena2a setup` no longer crashes on a corrupted `auth.serverUrl`.** The `isCloud` line called `new URL(auth.serverUrl).host` without a try/catch — a malformed file would crash the post-success path AFTER the agent was registered on the server, leaving server-client state divergence. Wrapped defensively; `isCloud` degrades to `false` on parse failure.
- **Help/error strings stop leaking the backend host.** `opena2a login --server <url>` help and the "server URL required" error message both used to say "omit for aim.oa2a.org" — that domain is for backend/API traffic, never for human eyes. Now mentions `aim.opena2a.org` (the user-facing domain) and explicitly calls out `localhost:8080` as the self-hosted shorthand.

### What just happened
`opena2a setup` text output expanded to include a "What just happened" block (agent registered with Ed25519 keypair, N MCP servers discovered/attached, trust score reflects the agent + every MCP it depends on) and a "Next:" block with concrete follow-up commands (`watch`, `identity list`, `trust <package>`). The cloud-auth path also prints a "Self-hosted instead?" hint that names a `--server` example so users on AIM Cloud are aware they can swap to their own AIM at any time.

## 0.10.0

### New Features
- **`opena2a review` HMA tab now consumes HackMyAgent Finding v2 (`evidence` / `rationale` / `concept` / `attackClass`).** HMA shipped the v2 schema in `dc8d344` (PR #143) with `attackClass` populated on every static-check finding (PR #146) and the evidence-driven Verify generator landing in PR #149. The `HmaFinding` interface in `packages/cli/src/commands/review.ts` now mirrors the four optional fields, and the JSON-parse boundary at the top of `runHmaPhase` calls a new exported `mapRawHmaFinding(raw)` pure helper that uses the `isHmaEvidence` / `isHmaRationale` runtime guards before passing values through. Older HMA builds without v2 fields still parse cleanly; the renderer falls back to the legacy `guidance` text and the curated `legacyRiskKb` entries.
- **HMA tab "Why this matters" prefers `rationale.plainEnglish` and renders structured evidence inline.** The HTML report's HMA findings table now shows the v2 rationale text first (with the legacy guidance / `legacyRiskKb` text as a fallback when no rationale is present), and renders a positive-evidence panel with the cited line numbers + verbatim content + per-line `why` notes when `evidence.kind === 'positive'`. Absence-of-defense evidence renders an "Observed but missing defense" panel with HMA's `observed.summary`. The `attackClass` label appears as a subtle inline tag next to the category column so CISOs can see the attack taxonomy without expanding a row.
- **HTML escaping tightened on the rationale / guidance render path.** The legacy guidance text and `legacyRiskKb` entries are now passed through `esc(...)` like every other user-facing string in the HMA tab. Previously they were emitted as raw HTML — a low-risk latent issue because HMA's emit sites produce plain text today, but tightening the contract closes the door on future regressions. Score-jump classification: category (a) preserved-detection / display-only (no detection narrowing).

### Bug Fixes (carried from the audit-bundle)
- **Credential scanner: skip test fixtures, demo scripts, and VHS recordings (audit B12 + B1).** The 2026-04-29 self-audit of `opena2a review` surfaced 7 credential findings on opena2a/main where every match was an intentional placeholder — three in `packages/aim-core/src/dlp/dlp.test.ts` (the DLP scanner's own fixtures), three in `docs/vhs/setup-lab.sh` and `scripts/demo-setup.sh` (terminal-recording demos), and one generic key in the same demo path. `opena2a protect` would have rewritten them and broken the DLP test suite. `walkFiles` in `packages/cli/src/util/credential-patterns.ts` now skips files whose basename matches `*.{test,spec}.{ts,tsx,js,jsx,mjs,cjs,py}`, `*_test.{go,py}`, or `demo[-_]*.{sh,ts,js,py}`, plus the `vhs/` directory segment. Production source files, non-demo shell scripts, and credential-bearing files outside those naming conventions still scan as before. Tradeoff documented inline: real credentials committed inside test/demo paths won't be flagged by this scanner — that's the wrong layer to defend against; use git pre-commit hooks instead. 16 unit tests (`packages/cli/__tests__/util/credential-patterns-fixture-exclusion.test.ts`) pin the regex behaviour and reproduce the audit's 7-finding-to-zero scenario.
- **`opena2a review` HMA tab labels (audit B3).** "Total Checks / Failed / Passed" displayed `60 / 60 / 0` because `runHmaPhase` derived all three counters from `parsed.findings` (HMA's failed-only array), making "0 Passed" structurally inevitable. HMA also emits `parsed.allFindings`, which carries every check that ran (passed and failed). New exported helper `deriveHmaCounts(parsed, failedCount)` reads `allFindings` when present and falls back to the failed count for older HMA versions. Self-scan now reads `152 Total / 48 Passed / 60 Failed`. 6 unit tests in `packages/cli/__tests__/commands/review-hma-counts.test.ts` cover both code paths.
- **`opena2a check --help` no longer claims "HMA + NanoMind" analysis (audit B17).** The string at `packages/cli/src/index.ts:251` advertised a NanoMind code path that does not exist in the cli today (no `@nanomind/*` dependency in `packages/cli/package.json`; daemon 0.2.0 ships separately and the cli wiring is soft-gated on telemetry per [CHIEF-CDS-033]). Now reads "full HMA security analysis runs". Will revert when daemon consumption lands.
- **`opena2a review` Shield phase: gate findings by scan target (closes #109 sub-item 1, PR #112).** SHIELD-INT-001 "Configuration file tampered" surfaced 288 critical occurrences on an empty target because Shield's event log carried state from prior scans of unrelated paths. New `filterEventsToTarget` helper narrows ConfigGuard events to `targetDir` before classification; other event sources (notably ARP) pass through to preserve `/usr/bin/curl` and similar absolute-path detections. 24 deterministic unit tests in `packages/cli/__tests__/shield/findings-path-scoping.test.ts`. Empty-test reproducer: composite 27→72.

- `guard harden` subcommand: scan skills for security issues via HackMyAgent, with `--fix` and `--dry-run` flags.

## 0.9.1

### Bug Fixes
- **`--no-contribute` per-invocation override (closes #107).** The flag was advertised in `--help` and the README telemetry table since 0.9.0 but never registered with Commander, so passing it produced `unknown option '--no-contribute'`. Now declared via `.option('--no-contribute', ...)`, threaded through the four `dispatchCommand` call sites in `index.ts` as `noContribute: globalOpts.contribute === false`, and the contribution gate in `router.ts` is rewritten to `(globalOptions.contribute || await isContributeEnabled()) && !globalOptions.noContribute` so the flag beats both `--contribute` and the persisted user-config consent for that single invocation. `RunOptions.noContribute?: boolean` added to `adapters/types.ts`.
- **Propagate `--no-contribute` and `--contribute` to spawned scanners.** `dispatchCommand` already injected `--format`, `--deep`, `--analm`, and `--static-only` into `adapterArgs`, but `--no-contribute` was being dropped before reaching `hackmyagent secure`, so the underlying subprocess still queued an anonymized summary even when opena2a-cli's own gate was closed. The flag now propagates so end-to-end suppression works (`opena2a scan --no-contribute` is silent; default and `--contribute` still queue). Independently rediscovered as P2-1 by the 0.9.0 `/release-test` walkthrough — strong signal the disclosure-without-implementation gap was real.

### Changed

- CommonJS / ESM bridge: the runtime imports of `@opena2a/telemetry` and `@opena2a/cli-ui` use dynamic `import()` inside the async `main()`, with `import type ... with { 'resolution-mode': 'import' }` for the TS type-only reference. Same pattern shipped in DVAA 0.8.2 and secretless-ai 0.16.0.
